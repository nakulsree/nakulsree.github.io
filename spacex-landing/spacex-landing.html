01 / Overview

Executive Summary

The critical cost driver behind SpaceX's $62M Falcon 9 launch price — roughly 60% cheaper than competitors — is first-stage booster reuse. If the first stage lands successfully, it can be refurbished and relaunched. If it doesn't, that unit is a write-off. Knowing in advance whether a landing is likely enables both operators and competitors to make informed pricing and mission planning decisions.

This project builds an end-to-end data science pipeline that answers this question: given mission parameters available before launch, can we predict whether the Falcon 9 first stage will successfully land? Data is collected directly from the SpaceX REST API and supplemented via Wikipedia web scraping; exploratory analysis is conducted both in Python and SQL; geospatial patterns are visualized with Folium; an interactive Plotly Dash app allows live filtering; and four classification algorithms are compared via 10-fold cross-validated GridSearchCV.

All four models converge to identical test accuracy of 83.3%, with the Decision Tree achieving the highest cross-validation accuracy of 88.9%. The binary target (Class: 1 = successful landing, 0 = failed) was engineered from raw outcome strings in the original dataset.

💰
Why This Matters
SpaceX charges $62M per Falcon 9 launch. Competitors charge $165M+. The difference is reusability. A landing-outcome model enables cost estimation, competitive analysis, and mission risk assessment — all before liftoff.
🚀
Data Source
SpaceX public REST API (v4) enriched with Wikipedia BeautifulSoup scraping. All API calls use helper functions to resolve nested IDs across rockets, launchpads, payloads, and cores endpoints.
🎯
Target Variable
Binary Class label engineered from raw Outcome strings: 1 = successful landing (True Ocean, True RTLS, True ASDS, None None), 0 = failed landing. Class mean ≈ 0.67 — roughly two-thirds of launches resulted in a successful recovery.

03 / Insights

Key EDA Findings

📈

Experience Drives Recovery

Flight number is a strong predictor of landing success — the scatter of Class vs. FlightNumber shows a clear upward trend. As SpaceX accumulated operational experience, booster recovery reliability improved substantially, independent of payload mass.

🌍

Orbit Type Is the Strongest Signal

Success rates vary dramatically by orbit type. Certain orbits (SSO, HEO, and GTO at high inclinations) present fundamentally harder recovery conditions due to energy constraints, while LEO and ISS missions have historically higher first-stage return rates.

🏗️

Site Capacity Specialization

VAFB SLC-4E handles no payloads exceeding 10,000 kg — consistent with its polar orbit specialization. KSC LC-39A handles the full payload spectrum including the heaviest missions. This site-payload relationship is a learnable signal for the classifier.

🗺️

Coastal Proximity is Universal

All four active launch sites (CCAFS LC-40, VAFB SLC-4E, KSC LC-39A, CCAFS SLC-40) are located within approximately 20 km of the coastline. This is both a safety requirement (debris over water) and a logistics constraint reflected in the geospatial analysis.

💡

SQL Milestone: First Ground Pad Landing

The SQL EDA identified the date of SpaceX's first successful ground pad landing using a simple MIN(Date) query filtered on landing outcome — a key historical milestone in the progression from drone ship to RTLS recovery capability.

⚖️

Class Imbalance: 67% Success Rate

The mean of the Class variable is approximately 0.67, meaning roughly two-thirds of recorded launches resulted in a successful first-stage recovery. This moderate imbalance favors predicting success but doesn't require aggressive resampling — standard model training handles it well.

Key Takeaway: All Models Converge at Test Time

All four classifiers achieve exactly 83.3% on the 18-sample test set despite varying CV accuracy (84.6%–88.9%). This is a consequence of the small test set size — 18 samples means each misclassification moves accuracy by ~5.6%. The CV accuracy spread (3.3 percentage points) provides a more reliable signal, where the Decision Tree's 88.9% suggests it captures genuinely more of the training structure. A larger test set would likely reveal more differentiation.

05 / Reflections

What I Learned

🔌

REST API Data Collection at Scale

The SpaceX API returns deeply nested JSON with IDs rather than names — every rocket, launchpad, payload, and core requires a separate API call to resolve. Building helper functions that abstract this resolution loop is the right pattern: it keeps the main data pipeline clean and makes re-querying individual fields straightforward without touching the main logic.

🕸️

Web Scraping as API Complement

When the REST API doesn't cover all historical records, web scraping fills the gap. BeautifulSoup HTML table parsing requires understanding the nesting structure of individual cells — not just find_all('tr') but also handling rowspan, Unicode normalization, and irregular cell layouts. The Wikipedia Falcon 9 table is particularly irregular and required custom helpers per column type.

🗃️

SQL as an EDA Tool in Jupyter

Using %sql magic with SQLite in a Jupyter notebook makes SQL a first-class EDA tool alongside Python. Loading a DataFrame into SQLite via to_sql() and then querying it with standard SQL aggregations — SUM, AVG, MIN, GROUP BY — is a practical pattern for analysts who think in SQL. The prettytable integration also produces clean tabular output inline.

🗺️

Geospatial Context Adds Interpretive Value

A Folium map of success/failure outcomes makes patterns immediately visible that aren't apparent from tables — the coastal clustering of all launch sites, the geographic separation between East and West coast facilities, and the density of KSC LC-39A missions. Visual context often surfaces hypotheses that lead to better feature engineering.

📉

Small Test Sets Mask Model Differences

All four classifiers achieving exactly 83.3% on the test set (18 samples, ~3 wrong each) is a practical reminder that test accuracy on small held-out sets is a coarse metric. CV accuracy — averaged over 10 folds — is a more reliable performance estimate here. This experience reinforced the value of reporting both metrics and interpreting them in light of dataset size.

🧮

Strong Regularization as a Signal About Features

Logistic Regression's best C=0.01 (strong L2 penalty) is interpretable: it says the classification boundary is relatively simple and most feature variability is noise. When the best-performing regularization is very strong, it's worth re-examining whether all features are truly informative — a prompt for more deliberate feature selection in future iterations.
