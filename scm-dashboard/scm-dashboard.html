01 / Overview

Executive Summary

This project is a full-stack web application designed to give supply chain operations teams analytical visibility across their entire network. Built as a team of six at Purdue University, it integrates data from multiple supply chain entities — manufacturers, distributors, and retailers — into a unified dashboard with role-specific views and real-time KPI calculations.

Two distinct user roles drive the design. Supply Chain Managers access operational views: company search and drill-down, shipment transactions, distributor performance by route, and disruption monitoring with active alerts. Senior Managers access strategic views: financial health scoring across all companies and regions, quarterly trend analysis, and a criticality scoring algorithm that identifies which companies pose the highest systemic risk if disrupted.

All database interactions use PDO prepared statements, enforcing parameterized query execution throughout and preventing SQL injection at every input boundary. XSS prevention via htmlspecialchars() is applied consistently at all output points.

Multi-Tier Network

03 / Architecture

System Architecture

The application is organized around two role-based portals, each with dedicated pages for different analytical tasks.

Supply Chain Manager
Operational visibility — real-time monitoring, company drill-downs, and transaction ledger.
Overview
KPI cards: total companies, total shipments, on-time delivery rate, active disruption count — computed live from the database.
Company Explorer
Searchable, filterable table (name, type, tier, region) with clickable rows linking to per-company detail views. On-time rate computed per company via JOIN + CASE WHEN aggregation.
Company Detail
Deep dive per company: upstream/downstream dependency graph (DependsOn), factory utilization (Manufacturer), route table (Distributor), product mix with volume share, full transaction history, delay histogram, financial health trend line, and disruption event log.
Disruptions
8 analytics modules: disruption frequency per company (events/month), recovery time histogram, high-impact disruption rate, total downtime, regional heatmap (2D: region × month), severity distribution, and active alert table for unresolved events.
Transactions
Unified ledger via UNION ALL across Shipping, Receiving, and InventoryAdjustment tables. Supports 6 simultaneous filters. Aggregate metrics recalculated on each filter change.
Distributor Details
Per-distributor analytics: shipment volume over time, on-time rate by route, status distribution, and disruption exposure score per route. Supports route and product sub-filters.
Senior Manager
Strategic oversight — financial health trends, regional risk, and network criticality scoring.
Financial Overview
Average financial health score by company, by type (Manufacturer / Distributor / Retailer), and by region. Quarterly trend line for any selected company. Year range filter updates all views simultaneously.
Disruptions & Risk
Regional disruption overview (stacked bar: high-impact vs. other), criticality scoring (downstream partners × high-impact events), event-level affected company drill-down, recovery time by region and category, disruption exposure score by region, and top-affected company ranking.
Companies & Logistics
Top distributors by shipment volume, per-distributor route and shipment detail, delay ranking with on-time percentage, and an add-company form that inserts into both the Company table and the appropriate subtype table (Manufacturer/Distributor/Retailer) in a single transaction.
04 / SQL

SQL Design Highlights

All queries use PDO prepared statements with positional parameter binding. Dynamic WHERE clauses are constructed programmatically and params are accumulated in order to match the SQL string.

UNION ALL — Unified Transaction Ledger

The transactions page unifies three structurally different tables (Shipping, Receiving, InventoryAdjustment) into a flat ledger using UNION ALL. Each branch computes a normalized Status column via CASE WHEN on date comparisons, and a DelayDays column using EXTRACT(DAY FROM (ActualDate::timestamp - PromisedDate::timestamp)). The outer query then applies a status filter and aggregation — requiring careful parameter ordering across three independent WHERE clauses before the outer status param.

SELECT Date, CASE WHEN ActualDate <= PromisedDate THEN 'Delivered' ... END AS Status
FROM Shipping s JOIN ... WHERE s.PromisedDate BETWEEN ? AND ?
UNION ALL
SELECT ReceivedDate, CASE WHEN ReceivedDate <= PromisedDate ...
FROM Receiving r JOIN Shipping s ...
UNION ALL
SELECT AdjustmentDate, 'Adjustment' AS Status FROM InventoryAdjustment
Criticality Score — Subquery JOIN

The Senior Manager disruptions page computes a criticality score for each company as the product of two independently aggregated subqueries: the number of downstream companies in the DependsOn graph, and the number of high-impact disruption events in the selected date range. Companies with high downstream reach AND high disruption exposure score highest — identifying systemic risk nodes in the supply network.

SELECT c.CompanyName,
  COALESCE(ds.DownstreamCount, 0) * COALESCE(hi.HighImpactCount, 0) AS Criticality
FROM Company c
LEFT JOIN (SELECT UpstreamCompanyID, COUNT(*) FROM DependsOn GROUP BY ...) ds
  ON c.CompanyID = ds.CompanyID
LEFT JOIN (SELECT AffectedCompanyID, COUNT(*) FROM ImpactsCompany
  WHERE ImpactLevel = 'High' AND EventDate BETWEEN ? AND ? ...) hi
  ON c.CompanyID = hi.CompanyID
ORDER BY Criticality DESC
Disruption Exposure Score

Used across both the SCM and Senior Manager portals, the disruption exposure formula weights high-impact events double: Exposure = Total Disruptions + 2 × High-Impact Events. This is computed via a single aggregation using conditional COUNT with CASE WHEN, requiring no subquery. The result is used both as a sortable table column and as a Chart.js bar chart.

SELECT l.ContinentName,
  COUNT(DISTINCT de.EventID) AS TotalDisruptions,
  COUNT(DISTINCT CASE WHEN ic.ImpactLevel = 'High'
    THEN de.EventID ELSE NULL END) AS HighImpact,
  (TotalDisruptions + 2 * HighImpact) AS ExposureScore
FROM DisruptionEvent de
JOIN ImpactsCompany ic ... JOIN Company c ... JOIN Location l ...
GROUP BY l.ContinentName ORDER BY ExposureScore DESC
Dynamic WHERE Clause Construction

Most pages support multi-filter queries where filter inputs from GET parameters are optional. The pattern accumulates SQL condition strings and matching parameter values in parallel arrays, then implodes conditions and executes via PDO with positional binding. This avoids string interpolation and keeps every filter parameterized regardless of which combination is active — a clean approach to dynamic query building in PHP without an ORM.

$sql = "SELECT ... WHERE 1=1";
$params = [];
if ($searchName !== '') {
  $sql .= " AND c.CompanyName LIKE ?";
  $params[] = '%' . $searchName . '%';
}
if ($typeFilter !== 'All') { ... }
$stmt->execute($params);
Delay Statistics with STDDEV

The company detail page computes on-time rate, average delay, and delay standard deviation in a single query using STDDEV_SAMP() — a PostgreSQL aggregate function applied to EXTRACT(DAY FROM (ActualDate - PromisedDate)). A companion query buckets individual shipments into delay bins for a Chart.js histogram. This gives operators both summary statistics and the full distribution in one page load.

SELECT
  COUNT(*) AS TotalShipments,
  AVG(EXTRACT(DAY FROM (ActualDate::timestamp - PromisedDate::timestamp))) AS AvgDelay,
  STDDEV_SAMP(EXTRACT(DAY FROM (ActualDate::timestamp - PromisedDate::timestamp))) AS StdDelay
FROM Shipping s
WHERE s.SourceCompanyID = ? AND s.ActualDate IS NOT NULL
Regional Heatmap — 2D Pivot in PHP

The disruptions page renders a color-coded heatmap of disruption count by region × month. The SQL returns a flat (region, YYYY-MM, count) result set. PHP post-processes it into a 2D associative array keyed by region and month, which the HTML template iterates as a table with computed background colors — from white (0) through yellow (5–10) to red (15+). No pivot extension required: pure PHP reshaping of a flat aggregate.

SELECT l.ContinentName, TO_CHAR(de.EventDate, 'YYYY-MM') AS Month,
  COUNT(DISTINCT de.EventID) AS Count
FROM DisruptionEvent de JOIN ... JOIN Location l
GROUP BY l.ContinentName, TO_CHAR(de.EventDate, 'YYYY-MM')
ORDER BY l.ContinentName, Month
-- PHP: $heatmap[$region][$month] = $count
05 / Analytics

Key Analytics Features

⏱️

On-Time Delivery Rate

Computed via conditional aggregation: SUM(CASE WHEN ActualDate ≤ PromisedDate THEN 1 ELSE 0 END) / COUNT(*). Calculated at the network level (overview), per company (explorer), per distributor (logistics), and per route — all driven by the same pattern with different GROUP BY keys.

🚨

Disruption Frequency (Events/Month)

Rather than raw counts, the disruptions page normalizes by observation window: it computes the number of months between filter start and end dates in PHP, then divides total disruption count per company by that figure — giving a monthly disruption rate that's comparable across different date ranges.

🏦

Financial Health Scoring

FinancialReport records contain a HealthScore per company per quarter. The Senior Manager view aggregates by company, type, and region; a company detail chart renders the quarterly trend. Scores are classified as Healthy (≥75), Watch (50–74), or At Risk (<50) — thresholds enforced consistently in PHP across all display contexts.

🗺️

Recovery Time Analysis

Recovery time is computed as EXTRACT(DAY FROM (EventRecoveryDate - EventDate)). The disruptions page bins individual recovery times into a histogram (0–5, 5–10, 10–15, 15–20, 20+ days) — processed in PHP — and also computes average recovery time by both region and disruption category.

🔗

Upstream / Downstream Dependency

The DependsOn table stores directed dependency relationships. The company detail page executes two separate queries — one traversing UpstreamCompanyID to find who the company depends on, one traversing DownstreamCompanyID to find who depends on it — rendering both as linked lists that navigate to other company detail views.

📦

Distributor Disruption Exposure by Route

The distributor detail page computes an exposure score per route: it joins Shipping with ImpactsCompany on both the source and destination company IDs, counting total disruptions and high-impact events. The formula Total + 2 × High weights severity, producing a comparable risk ranking across all routes for a given distributor.

06 / Reflections

What I Learned

🛡️

Security Is a Design Decision, Not an Afterthought

Applying PDO prepared statements and htmlspecialchars() consistently across 10+ pages required establishing a discipline early — not patching individually. The pattern of accumulating SQL condition strings and params in parallel arrays made this manageable across dynamic multi-filter queries without ever resorting to string interpolation.

🧩

Dynamic Query Building Without an ORM

Building dynamic WHERE clauses manually exposed the trade-offs that ORMs abstract away: parameter ordering must match the SQL string exactly, positional binding means every new filter adds a param at a specific position, and UNION ALL queries multiply this complexity across branches. Understanding this mechanics is fundamental to debugging query-related bugs at the database layer.

📊

PHP-to-JS Data Bridging

Rendering server-side query results as Chart.js visualizations requires json_encode() to embed PHP arrays as JavaScript variables in a script block. This is a simple pattern, but it surfaces an important constraint: the full result set must fit in the page at render time — there's no lazy loading or streaming. For large datasets, this would require pagination or a separate AJAX endpoint.

🗄️

PostgreSQL-Specific Syntax Matters

Functions like DATE_TRUNC, TO_CHAR, EXTRACT, STDDEV_SAMP, and the ::timestamp cast operator are PostgreSQL-specific. Code written against these functions is not portable to MySQL or SQLite without rewriting. Knowing which features are standard SQL vs. dialect-specific is practically important for project portability.

👥

Collaborative Development Requires Shared Conventions

In a team of six, inconsistent naming (column names in lowercase from PDO vs. uppercase in SQL, mixed date format handling, varying filter state management) created integration friction. Establishing a shared config, consistent column name conventions, and a shared CSS baseline early would have reduced the time spent resolving page-level inconsistencies later.

🔄

Role-Based Design Changes What Data You Need

Building two separate dashboards for different roles — not just hiding UI elements, but designing entirely different query strategies — clarified how operational and strategic questions require different granularities of data. An operations user needs per-shipment detail; an executive user needs aggregated health scores and risk rankings. Conflating these into a single view serves neither user well.
